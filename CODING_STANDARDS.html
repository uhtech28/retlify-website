File & Module Structure
Where code lives and how modules are organised
ℹ️
Current structure to enforce backend/ai/ for all AI services · backend/routes/ for Express routes · backend/utils/ for stateless helpers · frontend/ for all client assets. Never mix concerns across layers.
Must One export per service file
Each file in backend/ai/ exports exactly one primary function or class. module.exports = { runProductStudio } not a bag of unrelated utilities.
Must Keep promptBuilder pure — no I/O
ecommercePromptBuilder.js must contain only pure functions. No require('fs'), no network calls, no side-effects. This makes it fast and fully testable.
Must Separate routes from business logic
Route files in backend/routes/ are thin — they validate input, call a service, and send a response. Never write business logic inside a router. All AI logic belongs in backend/ai/.
Should Co-locate fallback logic with primary logic
The fallback image chain (Pollinations → Picsum → SVG) lives inside imageGenerationService.js, not scattered across files. Each service owns its own fallback.
Nice Add a SKILL.md or README per ai/ module
A short comment block at the top of each service file explaining: what it does, input shape, output shape, and any external dependencies. The existing file headers in this project are a good model — keep them.

Naming Conventions
Files, functions, variables, and constants
✓ Do
  • camelCase for functions and variables: buildPromptSet, generateImages
  • SCREAMING_SNAKE for module-level constants: QUALITY_SUFFIX, CACHE_TTL_MS
  • _underscorePrefix for private/internal functions: _buildStudioPrompt
  • kebab-case for all file names: image-generator-demo.html
  • Descriptive async function names ending in action verbs: generateImages, runProductStudio
✗ Don't
  • Don't use single-letter names outside loop counters or math
  • Don't name a function after what it receives: handleData — name it after what it produces
  • Don't mix naming styles in the same file
  • Don't use abbreviated names that lose meaning: genImg → use generateImage
  • Don't re-use the same name for different shapes in different files

Async & Error Handling
How we handle async operations and failures gracefully
⚠️
Golden rule: the UI must never break. Every async call that touches an external service (OpenRouter, Pollinations, Puter, Picsum) must have a .catch() handler that returns a valid fallback value — never null or undefined.
Must All external calls wrapped in try/catch
Every call to OpenRouter, Pollinations, Puter, Picsum must be wrapped. The catch block must log the error and return a graceful fallback — never rethrow to crash the pipeline.
Must Use Promise.all for parallel image generation
All 4 image generation calls run in parallel via Promise.all(tasks). Never await them sequentially — it would 4× the wait time for the user.
// ✓ Correct pattern — parallel with individual fallback const tasks = variations.map(v => { const info = promptSet[v]; return _generateOne(info.prompt, info.seeds[0], v) .catch(err => { console.warn(`[ImageGen] fallback for ${v}:`, err.message); return { url: _picsumUrl(v), source: 'picsum', type: v }; }); }); const images = await Promise.all(tasks);
Must Log errors with service prefix tags
All console.error / console.warn calls must include a bracketed service prefix: [ProductStudio], [ImageGen], [AI Search]. Makes debugging fast.
Should Add timeout to external HTTP calls
Pollinations calls should have an explicit AbortController timeout (e.g. 15 seconds) so a stalled upstream doesn't hang the pipeline. Fall to next provider on timeout.
Should Return consistent result shapes
Every service returns the same object shape whether it succeeded or failed (with a success: boolean flag). The caller should never have to guess the shape based on whether an error occurred.

AI & Image Pipeline
Rules for prompts, providers, and the fallback chain
Must Always run the full 4-provider fallback chain
Order: Puter.js (primary) → Pollinations (secondary) → Picsum (tertiary) → inline SVG (static final). Never skip a level. Each level is tried only if the previous fails.
Must Build prompts from productName + category, never raw user text
Raw user input ("shirt") is enriched by the prompt builder before it hits any image API. The builder adds material, lighting, setting, and quality suffixes from CATEGORY_CONTEXT. Direct string concatenation to the image API is forbidden.
Must Each image variation gets a distinct prompt
Studio, Lifestyle, Model, and Editorial shots must have meaningfully different prompts — not just the same prompt with minor changes. Refer to the 4 _build*Prompt functions as the canonical templates.
// ✓ Correct — category-aware studio prompt function _buildStudioPrompt(productName, category, style) { const { ctx } = _buildBase(productName, category); return [ `${productName}, ${ctx.material}`, `ecommerce product photography, pure white background, studio lighting`, `centered composition, front view, clean edges`, QUALITY_SUFFIX, ].join(', '); } // ✗ Wrong — raw user input sent directly const url = `https://image.pollinations.ai/prompt/${userInput}`; // NEVER
Must Add onError fallback on every <img> tag
Every image tag in frontend HTML must include onerror="this.src='https://picsum.photos/512?random='+index". This is the last line of defence — a broken image icon must never reach the user.
Should Cache generated images per (productName, category, variationType)
The in-memory cache key is productName|category|variationType. TTL is 30 minutes. Check the cache before calling any image provider. Cap the cache at 500 entries.
Should Never use HuggingFace or Replicate
These were removed in v3 and must not be re-introduced. They require API keys and have rate limits. Only Puter.js, Pollinations, and Picsum are approved image providers.

API Design (Express)
Route contracts, rate limiting, and response shapes
Must Every route has a rate limiter
Use express-rate-limit on every POST endpoint. The product-studio endpoint uses studioLimit (5 req/min). Text generation uses genLimit (10 req/min). Never expose an endpoint without a rate limit — this is a free-tier system.
Must Return consistent JSON shapes on success and error
Success: {success: true, ...data}. Error: {success: false, error: string, details?: string}. Never return different top-level shapes — the frontend shouldn't need to check shape before reading data.
Must Validate all required inputs before calling services
Check productName is a non-empty string before calling productStudioService. Return 400 with a clear error message. Never pass raw req.body to a service without validation.
Should Use the /health endpoint as your integration test
Keep the GET /api/ai/health endpoint up-to-date. It must accurately report AI enabled status, image provider, and feature list. Run it first when debugging a broken environment.

Frontend Standards
HTML, vanilla JS, and the image generation UI
Must Always show a loading state during generation
Show skeleton loaders for all 4 image slots immediately when generation starts. Never leave the user staring at a blank grid. Disable the generate button during loading to prevent duplicate requests.
Must Use CSS variables for all theme values
Colors, spacing, radii, shadows — all go into :root{} CSS variables. Never hardcode #FFD23F directly in a style property. This enables dark/light mode toggling without a rewrite.
Must Validate productName client-side before any API call
If productName is empty or whitespace, show an inline validation message and block the request. Don't waste a rate-limited API call on an empty input. Show feedback within 200ms of submit.
Should Label each image with its variation type
Every image in the 4-image grid must display a visible label: Studio · Model · Lifestyle · Editorial. These labels help users understand what they're seeing and improve perceived quality.
Should Add hover zoom on product images
transform: scale(1.04) on :hover with a 0.3s ease transition gives a premium feel. overflow:hidden on the card container keeps it clean.
Nice Indicate which provider served each image
Show a small badge (Puter / Pollinations / Picsum) on each image to communicate system status and help debug provider failures in production.

Security & Safety
Content filtering, secrets management, and input sanitisation
Must All secrets in .env — never in source code
OPENROUTER_API_KEY and any other credentials must be in .env (in .gitignore). The .env.example file stays committed. Running git grep -r "sk-or-v1" should return nothing.
Must Run all productName inputs through isSafe() before generation
The BLOCKED_TERMS set in productStudioService.js must be checked before any AI call. Do not add loopholes. If a term is borderline, add it to the set — it is better to block and explain than to generate problematic content.
Must Sanitise prompt strings before URL-encoding
Strip HTML tags and control characters from prompts before they become URL parameters to Pollinations. Use encodeURIComponent() — never raw string concatenation into a URL.
Should Cap prompt length at 500 characters
Built prompts should be truncated at 500 chars before being sent to any image provider. Long prompts degrade result quality and can cause URL length errors. The quality suffix is the last thing added, so truncate the middle if needed.

Performance
Caching, parallelism, and avoiding waste
Must Never make duplicate API calls for the same product+variant
The _cache Map in imageGenerationService.js prevents this. Always check the cache before firing a generation request. If cache is hit, return fromCache:true in the response.
Must Keep the in-memory cache under 500 entries
The eviction logic deletes the oldest entry when the cache size exceeds 500. Never remove this guard — on a shared server, unbounded growth causes OOM crashes.
Should Run description and image generation in parallel
Promise.all([generateProductDescription(...), generateImages(...)]) is the correct pattern. The text and images are independent — there is no reason to wait for text before starting image generation.
Nice Stream text response to frontend when possible
OpenRouter supports streaming. For long descriptions, stream the text to the frontend so users see content appearing in real time instead of waiting for the full response.

Git & Versioning
Commit messages, branching, and release labelling
Must Use conventional commit prefixes
feat: new feature · fix: bug fix · refactor: code change, no behaviour change · chore: tooling/config · docs: documentation. Example: feat(image-gen): add Puter.js as primary image provider
Must Never commit .env or node_modules
.gitignore must include .env, node_modules/, *.log, dist/, .DS_Store. Run git status before every commit — if you see any of these, stop and fix your .gitignore.
Should Tag releases with semantic versions
This project is currently at v5. Use git tag v5.1.0 for minor updates (new features), v5.0.1 for patch fixes. Breaking changes bump to v6.
Nice Keep a CHANGELOG.md at project root
One entry per release with: what changed, what was removed, what was fixed. The existing AI_UPGRADE_v2.md is a good model — centralise these into a single CHANGELOG.md going forward.
